# API修正 作業手順（2026-02-15）

## 前提
- 対象リポジトリ: `NaviAI-Back`
- 本手順は既存の実装（`GET /api/v1/users/me`, `GET /api/v1/dashboard`）を要件に合わせて修正する手順を示す。

## 目的
- `GET /api/v1/users/me` に管理者権限の有無を含める。
- `GET /api/v1/dashboard` に承認待ち件数と週次アクティビティ集計を追加する。

## 影響範囲（推定ファイル／箇所）
- コントローラ: `NaviAI-Back/app/src/main/java` 配下の `auth/controller/AuthController.java` または `user/controller/UserController.java`（`GET /api/v1/users/me` を提供している箇所）
- サービス: `service` パッケージ内の `AuthService`/`UserService` 実装
- DTO: `dto/UserResponse.java`, `dto/UserProfileResponse.java` など
- ダッシュボード: `summary/controller/SummaryController.java` または `dashboard` に相当するコントローラ
- リポジトリ: 承認待ちカウントやアクティビティ集計で利用する `Repository`（例: 投稿/記事テーブルのリポジトリ）
- テスト: `NaviAI-Back/app/src/test/java` 配下の対象テスト追加・更新

（※実際のファイル名・パスはリポジトリ内を確認の上、上記を置き換える。）

## 変更仕様（要約）
1. `GET /api/v1/users/me` のレスポンスに `isAdmin: boolean` を追加する。
   - 認可情報（ロール）を参照して判定する。
2. `GET /api/v1/dashboard` のレスポンスに以下を追加する。
   - `pendingApprovals`: 管理者が対応する承認待ち件数（整数）
   - `weeklyActivity`: 直近 N 週分の投稿/アクティビティ件数の配列（週開始日・件数 等のオブジェクト）

## 詳細作業手順

### 1. `users/me` の修正
1. 対象コントローラを開く（例: `UserController.java`）。
2. 返却している DTO（例: `UserResponse`）に `private boolean isAdmin;` を追加する。
3. サービス層で現在のユーザーの権限を判定し、DTO に `isAdmin` を設定する。
   - 例（擬似コード）: ユーザーのロール一覧に `ROLE_ADMIN` が含まれるかをチェック。

擬似実装例（変更箇所のイメージ）
- DTO 追加（`UserResponse.java`）:

```java
public class UserResponse {
    private Long id;
    private String username;
    private String email;
    private String displayName;
    private Instant createdAt;
    private boolean isAdmin; // 追加

    // getter/setter
}
```

- サービス側での設定（`AuthServiceImpl` / `UserServiceImpl` の該当メソッド）:

```java
UserResponse resp = map(user);
boolean admin = user.getRoles().stream().anyMatch(r -> "ROLE_ADMIN".equals(r.getName()));
resp.setIsAdmin(admin);
return resp;
```

4. コントローラのエンドポイント仕様に影響が出るため、OpenAPI/README 等があれば更新する。

### 2. `dashboard` の修正
1. コントローラ（例: `SummaryController.java`）のレスポンス DTO を拡張する。
   - 追加フィールド: `int pendingApprovals`, `List<WeeklyActivityDto> weeklyActivity`
2. 承認待ち件数算出
   - DB 条件 (例): `status = 'PENDING'` の件数をカウントする。対象テーブルは要件により異なる（例: `posts`, `submissions` など）。
   - JPA リポジトリ方法: `long countByStatus(Status.PENDING)` のようなメソッドを追加。

SQL イメージ:

```sql
SELECT COUNT(*) FROM posts WHERE status = 'PENDING';
```

3. 週次アクティビティ集計
   - 集計期間は直近 4 週程度を想定。
   - 投稿、コメント、いいねの日時の週単位でグルーピングして件数を集計する。

SQL イメージ（PostgreSQL の場合）:

```sql
SELECT date_trunc('week', created_at) AS week_start, COUNT(*)
FROM posts
WHERE created_at >= now() - interval '4 weeks'
GROUP BY week_start
ORDER BY week_start;
```

JPA/JPQL の場合はネイティブクエリで上記を実行するか、アプリケーション側で日付バケット化して集計する。

4. サービス実装で上記集計値を組み合わせて DTO を返却する。

## 付録: 注意点と推奨
- 承認待ち件数の集計対象テーブルやステータス値は要件定義（DB 設計）と合わせること。
- 集計クエリは頻度やデータ量によってパフォーマンス影響が出る可能性があるため、必要に応じてインデックス追加やキャッシュを検討する。
- API の契約変更（フィールド追加）はクライアントに通知し、リリースノートを作成すること。
