## API 仕様（ミドルウェア挙動）
- 認証失敗: 401 Unauthorized を返却（トークン未添付・期限切れ・改ざん等）
  - レスポンス例: HTTP 401, JSON body `{ "error": "Unauthorized", "message": "Invalid or missing token" }`
- 権限違反: 403 Forbidden を返却（role/permission 不足）
  - レスポンス例: HTTP 403, JSON body `{ "error": "Forbidden", "message": "Insufficient permissions" }`
- 正常: リクエスト継続、必要に応じて `SecurityContext` にユーザ情報/権限を設定。

運用上の決まり:
- 管理者系 API は `ROLE_ADMIN` を必須とする。
- 一般操作は `ROLE_USER` を最低限とし、細かい操作は `permissions` をチェックする。

## 実装方針
1. JWT 検証:
   - 既存の JWT 設定（署名鍵、アルゴリズム、期限）を `JwtUtils` に集約。
   - `JwtAuthFilter` で Authorization ヘッダ `Bearer <token>` を検証し、Claims から `roles` と `permissions` を抽出して `SecurityContext` に保存。
2. RBAC 方式:
   - コントローラ/メソッド単位で `@RequireRoles("ROLE_ADMIN")` / `@RequirePermissions("summary:read")` などのアノテーションを実装。
   - AOP（`RbacAspect`）で対象メソッド呼び出し前に `SecurityContext` の権限をチェック。
   - フィルタレイヤで既に拒否できる場合は早期に 401/403 を返す。
3. エラーハンドリング:
   - 認証エラー: 401
   - 権限エラー: 403
   - JSON で統一したエラー構造を返却（例: `{error,message,timestamp}`）
4. テスト:
   - `JwtUtils` の単体テスト（有効/無効トークン、期限切れ、改ざん）
   - `JwtAuthFilter` の単体テスト（認証成功/失敗、SecurityContext 設定）
   - `RbacAspect` の単体テスト（ロール/パーミッションがある場合と無い場合）

## 詳細な実装手順
1. `JwtUtils` を作成
   - 既存設定（`application.properties` 内の鍵や有効期限）を読み込む。
   - `validateToken(String token) : boolean`、`getAuthentication(token) : UsernamePasswordAuthenticationToken`、`getClaims(token)` を提供。
2. `JwtAuthFilter` を作成
   - `OncePerRequestFilter` を継承。
   - `doFilterInternal` で Authorization ヘッダ取得 → `JwtUtils.validateToken` → 成功時は `SecurityContextHolder.getContext().setAuthentication(...)` を設定。
   - 失敗時は `HttpServletResponse` に 401 を返却してリクエストを終了。
3. `@RequireRoles` / `@RequirePermissions` アノテーション作成
   - 注釈パラメータに複数指定可能な配列を許容。
4. `RbacAspect` (または `MethodInterceptor`) 実装
   - アノテーションが付いたメソッドの前で `SecurityContext` 内の `Authentication` を読み取り、roles/permissions を照合。
   - 不足時は `AccessDeniedException` を投げる（グローバル例外ハンドラで 403 にマッピング）。
5. グローバル例外ハンドラの確認/追加
   - `GlobalExceptionHandler` に `AccessDeniedException` をキャッチして 403 を返す処理を追加（既に存在する場合は確認）。
6. 単体テストの作成
   - `JwtUtilsTest`: 有効/無効/期限切れ/改ざんシナリオを検証するテストケース（各ケースに対して期待する boolean と例外動作をコメントで明記）。
   - `JwtAuthFilterTest`: MockMvc / MockHttpServletRequest を使って、Authorization ヘッダ有り無し、無効トークン、正常トークンでの挙動を検証（コメントで確認ポイントを明記）。
   - `RbacAspectTest`: モックの `Authentication` に roles/permissions を注入して、アノテーションで許可/拒否されることを検証。

## 単体テストの実行手順（Docker）
- テストは `NaviAI-Back` ディレクトリで実行する。以下のコマンドを使用する:

```powershell
cd NaviAI-Back
docker compose -f docker-compose.test.yml up --build
```

- 上記でコンテナが立ち上がり、テストが実行される想定。ログに失敗が出た場合は原因を修正して再実行する。
